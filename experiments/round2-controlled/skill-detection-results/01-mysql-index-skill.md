## Step 2: 命题识别

对每个核心概念，识别其在本文中**实际传达的命题**（读者会带走的关键洞察）：

1. **索引的本质**
   - 命题："索引是数据库的'目录'，通过预先构建的键值+地址映射，避免全表扫描。"
   - 读者洞察：索引的核心价值是用空间换时间，加速数据定位。

2. **索引的适用场景**
   - 命题："查询为主的字段应加索引，写操作为主/数据量小/重复值高的字段不应加索引。"
   - 读者洞察：加索引需要判断场景，不是所有字段都适合。

3. **索引的优缺点**
   - 命题："索引提升查询速度但增加存储成本和写入开销。"
   - 读者洞察：索引是读写权衡。

4. **主键索引**
   - 命题："主键索引是非空+唯一的约束，一张表只能有一个，通过 ALTER TABLE 创建。"
   - 读者洞察：主键索引是一种特殊的单列索引。

5. **唯一索引**
   - 命题："唯一索引要求字段值唯一但允许 NULL，通过 CREATE UNIQUE INDEX 创建。"
   - 读者洞察：唯一索引是另一种单列索引类型。

6. **普通索引**
   - 命题："普通索引无约束，仅提升查询速度，是最常用的索引类型。"
   - 读者洞察：普通索引是最基础的索引形式。

7. **多列索引/复合索引**
   - 命题："复合索引需遵循最左匹配原则，查询条件必须包含索引第一个字段。"
   - 读者洞察：复合索引有使用规则限制。

8. **二叉搜索树**
   - 命题："二叉搜索树每个节点最多2子节点，左小右大。"
   - 读者洞察：这是索引底层结构演进的起点。

9. **AVL 树**
   - 命题："AVL 树在 BST 基础上强制平衡，但节点仅存单个数据，层级仍过高。"
   - 读者洞察：平衡性解决了 BST 的退化问题，但单数据节点导致 I/O 多。

10. **BTree（B 树）**
    - 命题："B 树是多叉树，内部节点和叶子节点都存数据，但不支持高效范围查询。"
    - 读者洞察：多叉降低了层级，但范围查询是短板。

11. **B+ 树**
    - 命题："B+ 树内部节点只存键值、叶子节点存完整数据且通过链表连接，层级少、范围查询高效、查询稳定。"
    - 读者洞察：B+ 树是 MySQL 索引的最优底层结构，其设计直接决定了查询性能特征。

## Step 3: 替换实验结果

对每个命题执行替换实验：**如果这个命题被替换为另一种实现方式，用户的可观测行为是否改变？**

### 1. 索引的本质

- **替换**：将"索引是目录，通过键值+地址映射避免全表扫描"替换为"索引是哈希表，通过哈希值直接定位"
- **行为变化**：如果索引真的变成纯哈希结构，范围查询（BETWEEN、>、<）将无法高效执行，用户的查询模式需要根本性改变。
- **判定**：🟢 **Architectural** — "索引通过预构建的有序结构加速数据定位"是决定查询行为的核心机制。但文章仅停留在"目录"类比层面，未深入其结构性含义。

### 2. 索引的适用场景

- **替换**：将"查询为主的字段加索引，写操作为主不加"替换为"所有字段都加索引"
- **行为变化**：是。用户会为所有字段建索引，写入性能急剧下降，系统行为根本改变。
- **判定**：🟢 **Architectural** — 这是读写权衡的核心决策框架，决定了用户的索引设计行为。但文章仅以列表形式简要罗列，未深入阐述权衡逻辑。

### 3. 索引的优缺点

- **替换**：本质上是适用场景的另一种表述，同属"读写权衡"命题。
- **判定**：🟢 **Architectural** — 与适用场景重叠，是同一核心机制的不同切面。

### 4. 主键索引

- **替换**：将"主键索引（非空唯一）"替换为"主键使用普通索引+应用层唯一约束"
- **行为变化**：仅配置/选择变化。数据库仍能通过索引加速查询，用户的查询行为不变。
- **判定**：🔵 **Configurable** — 主键索引是 B+ 树上的一个配置选项（带 NOT NULL + UNIQUE 约束），其底层机制与普通索引相同。

### 5. 唯一索引

- **替换**：同理，替换为"普通索引+应用层唯一校验"
- **行为变化**：仅配置变化。
- **判定**：🔵 **Configurable** — 唯一索引是 B+ 树上的约束配置。

### 6. 普通索引

- **替换**：普通索引就是 B+ 树索引的默认形态。
- **判定**：🔵 **Configurable** — 是 B+ 树索引的基础配置。

### 7. 多列索引/复合索引

- **替换**：将"最左匹配原则"替换为"复合索引可从任意列开始匹配"
- **行为变化**：是。用户的查询写法、索引设计策略会根本改变。原本需要按最左列查询才能命中索引，替换后任何列组合都能命中。
- **判定**：🟢 **Architectural** — 最左匹配原则是复合索引的核心行为约束，直接决定了查询能否使用索引。但文章仅用一句话带过，严重不足。

### 8. 二叉搜索树

- **替换**：将"BST"替换为"哈希表"作为索引起点
- **行为变化**：否。这只是演进叙事的起点，不影响最终的索引行为。读者不需要知道 BST 就能理解 B+ 树。
- **判定**：🟠 **Transport** — BST 是 B+ 树演进叙事的背景素材。

### 9. AVL 树

- **替换**：将"AVL 树"替换为"红黑树"或其他平衡结构
- **行为变化**：否。演进叙事的中间步骤，不影响用户对 B+ 树的理解。

### 10. BTree（B 树）

### 11. B+ 树

- **替换**：将"B+ 树（内部节点只存键值、叶子链表连接）"替换为"LSM 树（写入优化的分层结构）"或"哈希索引"

- **判定**：🟢 **Architectural** — B+ 树的结构设计（内部节点仅存键值以降低层级、叶子链表以支持范围遍历、查询终止于叶子节点以保证稳定性）直接决定了 MySQL 索引的性能特征和可用查询模式。

## 额外发现

### 1. 三种索引类型的角色混淆

文章将主键索引、唯一索引、普通索引作为三个并列的核心概念呈现，但它们在架构层面是**同一机制（B+ 树索引）的不同配置**。真正的架构差异在于约束语义（NOT NULL、UNIQUE），而非底层实现。更合理的写法是：先讲 B+ 树索引机制（Architectural），再以简表说明三种类型的约束差异（Configurable），而非为每种类型独立成节。

### 2. 演进叙事的叙事重心陷阱

"从二叉树到 B+ 树"的演进叙事是技术文章中常见的写作模式，但容易造成**Transport 概念膨胀**——每一代结构都被赋予独立小节地位，而它们的实际作用仅是为 B+ 树的优越性做铺垫。更高效的做法是：将演进路径压缩为一张对比表或一段叙述，将省下的篇幅用于深入展开 B+ 树的结构细节（如节点分裂、页结构、Buffer Pool 交互）或最左匹配原则。

### 3. 命题粒度问题

B+ 树章节的命题在**概念层面**（"B+ 树层级少、范围查询高效、稳定性强"）和**实现层面**（"内部节点只存键值、叶子节点链表连接"）都有展开。按文章实际阐述深度，命题读为实现层面（文章给出了具体的结构设计说明），判定 Architectural。如果文章仅在概念层面提及"B+ 树是最佳选择"而未展开结构细节，命题应读为概念层面，仍判定 Architectural（因为即使在概念层面，"B+ 树决定索引行为"这一命题的替换也会改变用户行为）。

### 4. 文章结构与标题的承诺匹配度

文章标题承诺三件事："原理"、"分类"、"B+树底层逻辑"。实际权重分配：分类占 ~25%（含大量 SQL 语法），底层逻辑占 ~40%（但 ~15% 给了演进背景），原理仅在开篇 ~5% 简要提及。标题中的"原理"应对应更深入的 Architectural 阐述（如索引选择性、覆盖索引、索引下推等核心原理），而非仅停留在"目录"类比。当前文章更像是"分类 + 底层结构演进"而非"原理 + 分类 + 底层逻辑"。
