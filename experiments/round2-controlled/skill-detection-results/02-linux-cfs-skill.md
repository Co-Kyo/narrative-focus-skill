## Step 3: 替换实验结果

### 1. CFS vruntime 机制（被呈现为代码块，无解释）

- **替换**：将"vruntime + 红黑树"替换为"时间片轮转 + 链表"
- **行为变化**：**是**。CFS 的公平性保证、延迟敏感进程的响应性、权重（nice 值）对调度比例的影响——这些用户可观测的调度行为都会根本性改变。时间片轮转无法实现 CFS 的加权公平语义。
- **判定**：🟢 **Architectural** — vruntime 机制是 CFS 的核心架构机制，决定了进程如何被选择和调度，直接影响用户可观测的进程行为。

### 2. sched_class 多态接口

- **替换**：将"回调函数指针"替换为"switch-case 分发"或"直接函数调用"
- **行为变化**：否。调度器仍然能正确调度进程，用户行为不变。
- **判定**：🟠 **Transport** — 这是代码组织方式，不影响调度行为。

### 3. task_struct 字段

- **替换**：将字段名从 se/rt/dl 改为其他命名，或改用不同的存储结构
- **行为变化**：否。进程仍然能被正确调度。
- **判定**：🟠 **Transport** — 这是数据存储方式。

### 4. SCHED_FIFO API 调用

- **替换**：将 sched_setscheduler() 替换为 sched_setattr() 或通过 cgroup 设置
- **行为变化**：否。进程仍然能设置为实时调度。
- **判定**：🟠 **Transport** — API 是设置调度策略的手段，不是策略本身。

### 5. CFS sysctl 参数

- **替换**：将"sysctl -w kernel.sched_min_granularity_ns"替换为"echo > /proc/sys/kernel/sched_min_granularity_ns"
- **行为变化**：否。参数值相同，效果相同。
- **进一步分析**：但如果我们替换命题的粒度——"这三个参数控制 CFS 的调度行为"→ 替换为"CFS 没有可调参数，行为固定"→ 行为会变（无法调优延迟/吞吐权衡）。
- **判定**：🔵 **Configurable** — 这些参数是 CFS 机制的调节旋钮，不是机制本身。设置方式（sysctl vs procfs）是 Transport。

### 6. 创业视角类比

- **替换**：删除整个"创业视角分析"章节
- **行为变化**：否。读者对进程调度的理解不会减少任何技术内容。类比不传递任何独有的技术命题。
- **判定**：🔵 **Configurable**（甚至可以判定为"无技术价值的填充内容"）。不传递技术命题，不影响读者的技术心智模型。

### 7. 负载均衡

- **替换**：将"负载均衡"替换为"不做负载均衡，进程绑定固定 CPU"
- **行为变化**：**是**。多核系统中进程分布会根本性改变，某些 CPU 过载而其他空闲，影响系统整体吞吐和延迟。
- **判定**：🟢 **Architectural** — 负载均衡是多核调度的架构机制，决定了进程如何在 CPU 间分布。

### 8. 实时调度的优先级抢占模型

- **文章中未被阐述**。SCHED_FIFO、SCHED_RR、SCHED_DEADLINE 仅在开头列表中出现名称，标题承诺"到实时调度"但无任何解释。
- **替换**：将"实时调度严格按优先级抢占"替换为"实时调度按 nice 值公平分配"
- **行为变化**：**是**。实时任务的确定性保证会消失，硬实时场景会失败。
- **判定**：🟢 **Architectural** — 但文章完全没有阐述，属于缺失。

## 额外发现

### 1. 文章质量整体偏低

与典型的叙述重心错位问题不同，这篇文章的核心问题不仅是权重错位，更是**内容深度严重不足**。CFS 的 vruntime 机制本应是文章的核心（标题直接点了 CFS），但文章仅展示了数据结构定义，没有任何机制解释。这不是"Transport 被过度展示"的问题，而是"Architectural 内容根本没写"的问题。

### 2. "创业视角"章节的定位问题

"创业视角分析"将调度概念（资源分配、优先级、负载均衡、公平性）与企业管理类比。这种类比在入门科普文章中可以作为辅助理解工具，但前提是**技术内容本身已经充分阐述**。在本文中，技术内容极度薄弱，类比章节反而成了最大的独立章节（15%），形成了"技术没讲透，类比倒不少"的错位。

### 3. 代码示例的选择偏差

文章提供了三类代码示例：(1) 设置 SCHED_FIFO 的 C 代码，(2) 查看调度信息的 shell 命令，(3) sysctl 参数调优。这三类都是**操作层面**的示例，而文章缺失的是**机制层面**的示例——例如"展示 vruntime 如何随时间增长"、"展示红黑树如何选择下一个进程"、"展示不同 nice 值的进程如何获得不同 CPU 时间比例"。代码示例的选择进一步印证了文章将 Transport/Configurable 置于 Architectural 之上的错位。

### 4. 与同类文章的对比参考

一篇合格的 CFS 深度文章应至少涵盖：(1) vruntime 的计算公式（vruntime += delta_exec * NICE_0_WEIGHT / weight），(2) 红黑树的插入和选择逻辑（最左节点即下一个运行进程），(3) 调度延迟（sched_latency）和最小粒度（sched_min_granularity）如何影响调度行为，(4) nice 值到权重的映射关系。本文均未涉及。
