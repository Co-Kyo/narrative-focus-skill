## 额外发现

### 1. 命题粒度问题

**B-Tree 索引**存在明显的粒度歧义：
- 概念级命题："B-Tree 是 PostgreSQL 的默认索引类型，支持范围查询和排序" → Architectural
- 实现级命题："B-Tree 节点满时执行三步分裂算法" → Transport

文章选择了实现级粒度进行详述（节点分裂的三步算法），但这个粒度对"查询优化"主题没有贡献。按照 skill 方法论，命题粒度应由文章实际阐述深度决定——文章既然花大量篇幅讲节点分裂，就应该在这个粒度上评判，结果是 Transport。

### 2. 文章主题与内容的结构性错配

文章标题和目标定位是**查询优化**（如何让查询更快），但章节一和章节二的实际内容是**索引数据结构**（索引内部怎么实现的）。这是两个不同的知识域：
- 查询优化：执行计划 → 瓶颈识别 → 优化手段
- 索引原理：数据结构 → 实现算法 → 类型比较

文章将"索引原理"内容放在了"查询优化"的框架下，导致读者带着"如何优化查询"的期望来读，却获得了"索引内部怎么工作"的知识。

### 3. 方法论适用性说明

本文是一篇**教程型技术文章**（technical tutorial），属于 skill 的目标文章类型。替换实验在本案例中执行较为直接——技术细节与读者行为之间的因果链清晰。

但需注意：对于某些高度概念化的文章（如架构设计哲学），替换实验可能需要在更抽象的层面进行。本案例不存在此问题。

### 4. "被遮蔽"模式的典型性

本文是"Architectural 概念被遮蔽"的典型案例：多个真正决定读者优化行为的概念（EXPLAIN ANALYZE 使用、扫描方式选择、连接算法选择、统计信息机制）被压缩到极短的篇幅中，而 Transport 概念（索引内部实现）获得了大量独立阐述。这种模式在技术教程中常见——作者倾向于详述自己深入了解的底层机制，而将上层的诊断/决策框架一笔带过。
