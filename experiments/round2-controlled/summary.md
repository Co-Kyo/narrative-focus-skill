# Narrative Focus Skill — 第二轮对照实验汇总报告

> **实验时间**: 2026-04-24
> **实验模型**: 小米 MiMo v2.5-pro（两组子 agent 均基于同一模型，区别仅在于是否加载 Skill 输入）
> **实验设计**: 3 篇真实技术教程 × 2 组子 agent（无 Skill 对照组 vs Skill 组）= 6 份报告的对照分析
> **核心问题**: 在同一模型下，Skill 的方法论框架相比无 Skill 输入时的直觉判断，增量价值是什么？

---

## 二、对照结果总览

### 2.1 MySQL 索引文章

| 维度 | 无 Skill 对照组 | Skill 组 |
|------|----------------|----------|
| **检出问题数** | 4 个大类 | 8 个具体问题 |
| **错位定位** | "SQL语法权重偏高"、"B树演进权重偏高"、"聚簇/覆盖索引缺失"、"B+树深度不够" | 3 个 Configurable 被 Disproportionate + 3 个 Transport 被 Over-highlighted + 1 个 Architectural 被 Obscured |
| **角色判定** | 无（只说"权重偏高/不足"） | 每个概念有明确的 A/T/C 标签 |
| **核心发现** | 聚簇索引、覆盖索引、回表、索引失效——**但这些概念在文章中根本不存在** | 最左匹配原则——**文章中存在但被压缩的概念** |
| **改进建议** | 5 条建议，含新增章节结构 | 无修正流程（检测终止） |

**关键差异**：
- 无 Skill 对照组发现了文章**缺失的内容**（聚簇索引、覆盖索引等），但这些内容在原文中不存在——这是"缺失内容检测"而非"叙述重心错位检测"
- Skill 组严格限于**文章中存在的概念**的权重评估，检出了最左匹配原则被压缩、三种索引类型权重过高、B树演进叙事过度展开
- 无 Skill 对照组的"B+树深度不够"判断更精准（指出了缺结构图、缺具体示例），而 Skill 组只说"B+树权重正确"——**Skill 组对"正确但不够深"的情况缺乏评价能力**

### 2.2 Linux CFS 文章

| 维度 | 无 Skill 对照组 | Skill 组 |
|------|----------------|----------|
| **检出问题数** | 4 个大类（含 7 个具体缺失） | 6 个具体问题 |
| **核心发现** | 结构体代码搬运占 30%、创业类比无关、vruntime 完全缺失、实时调度未展开 | 3 个 Architectural 被 Obscured + 2 个 Transport/Config 被 Over-highlighted + 创业类比被标记 Disproportionate |
| **评分** | **1/5 分**（"基本不具备教学价值"） | 无评分（仅错位检测） |
| **额外发现** | 指出"代码示例与概念解释脱节"、"贴代码不等于讲原理" | 指出"核心概念根本没写"、"代码示例停留在操作层面而非机制层面" |

**关键差异**：
- **两者结论高度一致**：都认为 vruntime 机制是最大缺失，代码搬运和创业类比是主要问题
- 无 Skill 对照组给出了**量化评分**（1/5），Skill 组没有
- 无 Skill 对照组更擅长发现**文章缺失的核心概念**（调度时机、cgroup 调度、sched_latency 解释等），Skill 组更擅长**精确分类已有概念的错位类型**
- Skill 组额外识别出了"创业视角类比不传递技术命题"——这是替换实验的自然结论（删除类比不影响读者技术理解）

### 2.3 Git 底层对象文章

| 维度 | 无 Skill 对照组 | Skill 组 |
|------|----------------|----------|
| **检出问题数** | 4 个问题 | 2 个错位 |
| **核心发现** | tree object 被严重压缩、SHA-1 寻址缺失、packfile 缺失、tag 权重偏高 | tag object Over-highlighted + 删除大文件章节 Disproportionate |
| **对"轻度错位"的处理** | 识别出了 4 个问题，匹配度评估为 60% | 5 项正确、2 项错位——**更精确地量化了"大部分是对的"** |

**关键差异**：
- 这是一篇**轻度错位**的文章（大部分概念权重正确）
- Skill 组的优势在此最为明显：**精确识别出 5 项正确 + 2 项错位**，而无 Skill 对照组倾向于"找问题"，给出了 4 个问题但没有明确说哪些是对的
- 无 Skill 对照组发现了**缺失的内容**（SHA-1 寻址、packfile 结构、tree 内部结构），Skill 组只评估**已存在概念的权重**
- **两者的 tag object 判断完全一致**：都认为权重偏高

## 三、对照分析：Skill 的增量价值

### 3.1 Skill 组做得更好的地方

| 能力 | 说明 | 证据 |
|------|------|------|
| **结构化分类** | 每个概念有明确的 A/T/C 标签和 ✅/❌ 判定 | MySQL 的 11 个概念逐一分类，无 Skill 对照组只有模糊的"权重偏高/不足" |
| **精确量化** | 能区分"大部分正确 + 少量错位" vs "严重错位" | Git 文章：Skill 组报告 5 正确 + 2 错位；无 Skill 对照组只列了 4 个问题，没说哪些对了 |
| **错位类型细分** | Over-highlighted / Obscured / Disproportionate 三种错位模式 | 无 Skill 对照组只有"权重偏高"和"权重不足"两种 |
| **命题粒度分析** | 能识别同一概念在不同粒度下的不同判定 | B+ 树的概念级 vs 实现级命题分析，无 Skill 对照组无此能力 |
| **避免"缺失内容"噪音** | 严格限于文章已有概念的权重评估 | 无 Skill 对照组混入了"聚簇索引缺失"等不属于叙述重心错位的评价 |

### 3.2 无 Skill 对照组做得更好的地方

| 能力 | 说明 | 证据 |
|------|------|------|
| **缺失内容发现** | 能指出文章应该有但没有的核心概念 | MySQL：聚簇、覆盖索引、回表、索引失效；Linux：调度时机、cgroup；Git：SHA-1、packfile |
| **深度评估** | 能判断"写了但不够深" | MySQL："B+树部分深度不够，没有结构图或具体示例"；Skill 组只说"B+树权重正确" |
| **量化评分** | 给出整体质量评分 | Linux CFS 无 Skill 对照组给出 1/5 分 |
| **改进建议** | 提供具体的结构调整方案 | MySQL 无 Skill 对照组给出了完整的 7 章节重组方案 |
| **"填充内容"识别** | 更敏锐地识别低价值内容 | Linux CFS 无 Skill 对照组直接指出"创业视角是 CSDN 流量文章的填充段落" |

### 3.3 两组共有的能力

| 能力 | 一致性 |
|------|--------|
| **tag object 权重偏高** | ✅ 完全一致 |
| **创业类比无技术价值** | ✅ 完全一致 |
| **B 树演进叙事过度展开** | ✅ 完全一致 |
| **核心概念缺失识别** | ⚠️ 部分一致（无 Skill 对照组发现更多缺失） |

## 四、实验设计的反思

### 4.1 本次实验的改进

| 第一轮问题 | 本次改进 | 效果 |
|-----------|---------|------|
| 只有合成文章 | 使用真实技术教程 | 更贴近用户实际使用场景 |
| 无对照组 | Skill 组 vs 无 Skill 对照组（同一模型） | 能区分 Skill 的增量价值 |
| 错位幅度太大 | 选择错位程度不同的文章（Git 轻度、MySQL 中度、Linux 重度） | 更好地测试 Skill 在不同场景下的表现 |

### 4.2 本次实验仍未解决的问题

| 问题 | 说明 |
|------|------|
| **仍然没有测试前处理模式** | 3 篇文章都是后处理检测，前处理（收集阶段标注）未测试 |
| **仍然没有测试修正效果** | 只测了检测准确度，没测修正后的文章是否真的更好 |
| **仍然没有真实用户参与** | 没有"读者读完 before/after 后理解效果对比" |
| **样本量仍然不足** | 3 篇文章 × 2 组 = 6 份报告，统计上不具显著性 |
| **对照组可能不是真正的"无 Skill"状态** | 同一个 MiMo v2.5-pro 即使不加载 SKILL.md，也可能因为训练数据中接触过类似方法论而表现优于真正的"无先验知识"判断 |

### 4.3 核心发现：Skill 的定位应该是"精确标尺"而非"全面体检"

对照实验揭示了一个关键洞察：

**Skill 组的核心价值是"精确标尺"**——对已有概念做精确的角色分类和权重评估。它擅长回答"这个概念被放对位置了吗？"

**无 Skill 对照组的核心价值是"全面体检"**——发现缺失、评估深度、给出改进方案。它擅长回答"这篇文章缺了什么？写得够深吗？"

两者是**互补关系**而非替代关系。最佳实践应该是：
1. 先用 Skill 的替换实验做**精确错位检测**（找到已存在概念的权重问题）
2. 再用无 Skill 的直觉做**缺失内容审查**（找到应该存在但不存在的概念）
3. 最后结合两者给出**完整的改进建议**
