## Step 3: 替换实验结果

对每个命题执行替换实验：**如果该命题被替换为替代命题，用户可观测行为是否变化？**

### 概念 1：X.509 证书链验证

- **命题**："客户端通过证书链验证服务器身份（过期、吊销、签名链、域名匹配）"
- **替换为**："客户端通过 Trust-on-First-Use (TOFU) 模型验证服务器身份（首次连接记住指纹，后续连接比对）"
- **行为变化？** ✅ 是。用户将无法理解 CA 体系的撤销能力、企业 PKI 部署、证书透明度日志等核心安全行为。验证模型完全不同。
- **判定：Architectural** — 证书链验证是 TLS 信任模型的核心机制，决定"你连接的是谁"这一根本问题。

### 概念 2：加密套件列表

- **命题**："TLS 1.3 有 3 个推荐套件，TLS 1.2 套件命名格式为 KX_AUTH_ENC_MAC"
- **替换为**："TLS 1.3 有 5 个推荐套件，命名格式为 ENC_KX_MAC"
- **行为变化？** ❌ 否。用户读到的仍然是"有若干套件可选"，具体是 3 个还是 5 个、命名格式如何，不影响用户对 TLS 握手的理解或操作。这是**枚举/配置参考**。
- **判定：Transport（实为 Configurable）** — 套件列表是密钥协商和加密机制的配置选项，本身不决定系统行为。

> 注：严格来说，加密套件是 Configurable（"现有机制上的开关/选项"），但文章把它呈现为独立核心章节（20% 篇幅），其叙述权重远超其实际角色。

### 概念 3：Client Hello / Server Hello

- **命题**："握手第一步是 Client Hello，包含 TLS 版本、套件列表和随机数"
- **替换为**："握手第一步是 Client 发起连接请求，服务端返回能力清单"
- **行为变化？** ❌ 否。用户理解的是"双方先交换能力信息"这一流程步骤，具体字段名（Client Hello vs 连接请求）不影响对 TLS 安全机制的理解。这是**协议消息格式/传输细节**。
- **判定：Transport** — Client Hello 是承载协商信息的传输载体，不是决定安全行为的机制。

### 概念 4：密钥协商

- **命题**："TLS 1.3 使用 ECDHE 进行密钥交换"
- **替换为**："TLS 使用 RSA 密钥传输（客户端用服务器公钥加密预主密钥）"
- **行为变化？** ✅ 是。RSA 密钥传输不提供前向安全性（服务器私钥泄露可解密所有历史会话），而 ECDHE 提供。这是 TLS 安全模型的根本差异。用户若不知道密钥协商机制的区别，就无法理解为什么 TLS 1.3 废弃了 RSA 密钥传输。
- **判定：Architectural** — 密钥协商机制直接决定会话的安全属性（前向安全性、抗量子性等）。

### 概念 5：会话恢复

- **命题**："Session ID 和 Session Ticket 机制避免重复握手"
- **替换为**："每次连接都执行完整握手"
- **行为变化？** ✅ 是。完整握手 vs 恢复握手在延迟、计算开销上有显著差异，直接影响用户体验和服务器性能。这是影响系统可观测行为的机制。
- **判定：Architectural** — 会话恢复机制决定连接性能行为。

### 概念 6：前向安全性

- **命题**："每次会话使用独立的临时密钥"
- **替换为**："所有会话使用同一长期密钥加密"
- **行为变化？** ✅ 是。长期密钥泄露将暴露所有历史会话内容，而临时密钥确保单次会话泄露不影响其他会话。这是用户需要理解的核心安全属性。
- **判定：Architectural** — 前向安全性是 TLS 安全保证的核心组成部分。

## Step 5: 检测报告

- **[✅ 正确]** X.509 证书链验证：呈现为核心（30%），实际为 Architectural，权重匹配
- **[❌ 过度突出]** 加密套件列表：呈现为独立核心章节（20%），实际为 Configurable；套件枚举和命名格式是配置参考，不是决定握手安全行为的机制
- **[❌ 过度突出]** Client Hello / Server Hello：呈现为独立核心章节（15%），实际为 Transport；协议消息格式是承载协商信息的载体，不是安全机制本身
- **[❌ 被遮蔽]** 密钥协商（ECDHE）：被压缩为一句话（10%），实际为 Architectural 核心；密钥协商机制直接决定前向安全性等安全属性，应有独立深入展开
- **[❌ 被遮蔽]** 会话恢复：被压缩为名称提及（10%），实际为 Architectural；Session ID vs Session Ticket 的行为差异（无状态 vs 有状态、安全权衡）值得展开
- **[❌ 被遮蔽]** 前向安全性：被压缩为一句话（5%），实际为 Architectural；这是 TLS 1.3 的核心设计目标之一，与 ECDHE 密钥协商直接关联，不应被边缘化

**总结：6 个核心概念中，1 个正确，5 个存在叙述重心错位。** 文章将 Transport/Configurable 级别的细节（证书验证步骤的展开、套件列表的枚举、协议消息格式）过度突出为独立章节，同时将真正决定安全连接行为的 Architectural 机制（密钥协商、前向安全性、会话恢复）压缩为流程步骤中的一句话。这与文章标题"详解握手过程"的承诺形成显著偏差。

## 额外发现

### 1. 命题粒度问题：证书验证的展开深度争议

证书链验证在文章中被展开到**步骤级**（4 个验证步骤 + 完整 Python 代码 + Let's Encrypt 子节），这给了它极高的叙述权重。但替换实验显示其命题确实是 Architectural。问题在于：**证书验证虽然是 Architectural，但在"握手过程详解"这一标题下，它是否应该占据 30% 的篇幅？**

证书验证回答的是"你连接的是谁"，而密钥协商回答的是"你的连接有多安全"。两者都是 Architectural，但在"一次安全连接是如何建立的"这个语境下，密钥协商（如何建立安全通道）比身份验证（验证对方身份）更接近标题承诺的核心。证书验证的 30% 可以保留，但不应以牺牲密钥协商和前向安全性为代价。

### 2. 方法论适用性：协议教程中的"流程步骤"陷阱

TLS 握手是一个**时序流程**（Client Hello → Server Hello → Certificate → Key Exchange → Finished），SOP 的方法论在处理这类协议教程时有一个需要注意的模式：**流程步骤天然倾向于被写成等权的线性叙述**，导致真正决定安全属性的步骤（密钥协商）和纯粹的传输步骤（Client Hello 消息格式）被同等对待。

建议在检测此类文章时，额外关注：**流程中的哪些步骤承载了 Architectural 机制，哪些只是 Transport 载体？** 文章应将 Architectural 步骤从流程线中"抽出来"做独立深入展开，而不是让它们淹没在 "第一步…第二步…第三步…" 的线性叙述中。

### 3. 证书验证子节的过度展开

证书验证章节中，Let's Encrypt / ACME 协议子节（含 certbot 命令）对于"握手过程详解"这一主题来说属于**离题内容**。Let's Encrypt 是证书颁发机构的运维操作，不是握手过程的一部分。这一子节进一步拉高了证书验证的叙述权重，使其从"重要但合理的 Architectural 展开"变成了"挤占其他核心概念空间的过度展开"。

### 4. 总结段落与正文结构不一致

文章总结写道"TLS 握手通过证书验证和密钥协商建立安全连接"，但正文结构并未反映这一总结——证书验证占 30%，密钥协商仅占 10%。总结本身是合理的叙述重心定位，但正文结构与之严重脱节。
