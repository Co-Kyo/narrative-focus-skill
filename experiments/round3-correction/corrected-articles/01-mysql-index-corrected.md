# 拆解MySQL索引：原理、分类与B+树底层逻辑详解

> 来源：博客园 https://www.cnblogs.com/bxxr/p/19713949
> 收集时间：2026-04-24

索引是 MySQL 优化查询性能的核心手段，如同书籍的目录，能帮助数据库快速定位数据，避免全表扫描的低效操作。本文将从索引的基础概念出发，详解索引的分类、优缺点、创建方式，以及底层的 B + 树结构原理，帮你彻底理解索引的工作机制。

## 一、索引的核心概念

### 1.1 索引的本质

索引本质上是数据库的 "目录"，通过预先构建的目录信息（键值 + 数据地址），让数据库无需全表扫描（逐行逐字段匹配），就能快速找到目标数据。

无索引 vs 有索引：

- 无索引：查询数据时需全表扫描，数据量越大，查询越慢；

- 有索引：通过索引结构直接定位数据位置，查询效率呈数量级提升。

### 1.2 索引的适用场景

#### ✅ 建议加索引的场景

- 以查询（SELECT）为主的字段（如用户 ID、订单号、商品名称）；

- 经常作为 WHERE 条件、JOIN 关联、ORDER BY 排序的字段。

#### ❌ 不建议加索引的场景

- 以写操作（INSERT/UPDATE/DELETE）为主的字段：写入数据时，需同步更新索引的底层数据结构（如 B + 树），会显著降低写入性能；

- 数据量极小的表（如配置表，全表扫描比走索引更快）；

- 重复值极高的字段（如性别字段，索引筛选效果差）。

### 1.3 索引的优缺点

优点：显著提升查询速度、降低磁盘 I/O 次数（索引按字段排序）、避免服务器排序/临时表操作

缺点：增加存储成本（索引需额外磁盘空间）、降低写入性能（写入需同步更新索引）、索引可能失效（如查询条件与索引不匹配）

## 二、索引的分类与创建方式

MySQL 索引按字段数量可分为单列索引和多列索引。三种单列索引的核心区别如下：

| 类型 | 约束 | 特点 | 创建示例 |
|------|------|------|---------|
| 主键索引（PRIMARY KEY） | 非空 + 唯一 | 一张表只能有一个，定义主键时自动创建 | `ALTER TABLE test_index ADD PRIMARY KEY (id);` |
| 唯一索引（UNIQUE） | 唯一，允许 NULL | 保证字段值不重复 | `CREATE UNIQUE INDEX idx_phone ON test_index (phone);` |
| 普通索引（INDEX） | 无约束 | 最常用，仅提升查询速度 | `CREATE INDEX idx_price ON test_index (price);` |

多列索引（复合索引）遵循最左匹配原则——查询条件必须包含索引的第一个字段才能生效：

```sql
CREATE INDEX idx_id_name ON test_index (id, name);
```

## 三、索引的底层实现：从二叉树到 B + 树

MySQL 索引的底层默认采用B + 树结构，这是经过多代树形结构优化后的最优方案。理解这一演进过程，是理解索引工作原理的关键。

### 3.1 为什么需要更优的树结构

索引的核心需求是**快速查找**。树形结构天然适合这一场景，但不同的树结构在查找效率上差异巨大，而查找效率直接决定了磁盘 I/O 次数——这是数据库性能的瓶颈。

### 3.2 二叉搜索树（BST）

二叉搜索树是最基础的查找树：每个节点最多 2 个子节点，左子节点值小于父节点，右子节点值大于父节点。

**查找效率**：平均 O(log n)，但**最坏情况会退化为 O(n)**——当数据按顺序插入时，树会退化成链表，查找退化为全表扫描。

> 这是二叉搜索树的根本缺陷：无法保证稳定的查找效率。

### 3.3 AVL 树（平衡二叉树）

AVL 树通过**强制平衡**解决了退化问题：每个节点的左右子树高度差 ≤ 1，保证了 O(log n) 的查找效率。

**局限**：每个节点仅存储单个数据，当数据量大时树的层级仍然过高。对数据库而言，每一层树都对应一次磁盘 I/O，层级过高意味着 I/O 次数过多。

> AVL 树解决了稳定性问题，但未解决磁盘 I/O 效率问题。

### 3.4 B 树（BTree）

B 树是面向磁盘存储设计的多叉树，做了两个关键改进：

- **多路分支**：单个节点可存储多个键值，大幅降低树的层级
- **内部节点存数据**：内部节点和叶子节点都存储数据

**局限**：内部节点存储数据后，单个节点能容纳的键值减少，树的层级相对更高。且范围查询需要中序遍历整棵树，效率不高。

### 3.5 B + 树（MySQL 默认选择）

B + 树在 B 树基础上做了针对性优化，完美解决了上述所有问题：

**核心结构特点**：

- 内部节点仅存储键值（不存数据），单个节点能存储更多索引项，树的层级更少
- 叶子节点存储完整数据，且所有叶子节点通过链表连接
- 所有数据查询最终都要到叶子节点

**为什么 B + 树是最优选择**：

- **减少磁盘 I/O**：内部节点只存键值，单个节点能存储更多索引项，树的层级更少（通常 3-4 层即可支撑千万级数据）
- **范围查询高效**：叶子节点链表有序，范围查询只需遍历链表，无需回溯
- **查询稳定性高**：所有查询路径长度一致（都到叶子层），性能可预测

## 四、核心总结

- 索引核心价值：通过 "目录" 机制避免全表扫描，提升查询速度
- 底层结构选择：MySQL 索引默认采用 B + 树，相比二叉树、B 树，其层级少、范围查询高效、稳定性强
- 索引类型关键：主键索引（非空唯一）、唯一索引（唯一可空）、普通索引（无约束），都是 B + 树上的不同约束配置
