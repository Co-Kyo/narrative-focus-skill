## 检测摘要

原始文章检出 2 项叙述重心错位：

1. **❌ Disproportionate（过度突出）**：主键索引/唯一索引/普通索引 — 三个索引类型各自拥有独立小节 + 代码示例，占据第二节大量篇幅，但实际角色是 B+树索引机制上的约束配置（Configurable），不是决定系统行为的核心机制。
2. **❌ Obscured（被压缩）**：二叉搜索树 → AVL树 → BTree → B+树 的演进路径 — 每种树仅以 2-3 行 bullet point 呈现，实际角色是 Architectural，是读者理解"为什么 MySQL 选择 B+树"的核心叙事线。

### 叙事链完整性校验

修正后的文章建立了一条完整的因果叙事链：

```
为什么需要索引（避免全表扫描）
  → 索引的底层是树形结构
    → 为什么 BST 不够（退化为 O(n)）
      → 为什么 AVL 不够（层级过高，I/O 多）
        → 为什么 B 树不够（内部节点存数据，键值少；范围查询低效）
          → 为什么 B+树是最优解（内部节点只存键值 → 层级少；叶子链表 → 范围查询高效）
```

**判定**：✅ 叙事链完整，每一环的"局限 → 改进"因果关系清晰。读者沿此链路学习，能建立"为什么"的理解，而非仅记忆"是什么"。

本修正报告的评估维度聚焦于**叙述重心**，而非内容完备性。具体而言：

- **评估的是**：各概念获得的篇幅权重是否与其在"理解 MySQL 索引"这个目标下的实际重要性匹配
